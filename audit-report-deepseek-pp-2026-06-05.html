📝 总体评估

📋 总体评估

项目概况：DeepSeek++ 是一个 WXT + React + TypeScript 的 MV3 浏览器扩展，在 chat.deepseek.com 上通过 main-world fetch/XHR hook 注入记忆、Skill、MCP/本地工具能力，并通过 sidepanel 管理 memory、Skill、MCP、Automation、WebDAV 和 Shell Native Host。

主要风险：最大的风险不在构建链路，而在浏览器扩展隔离边界：main-world 与 content script 之间使用可伪造、可旁路监听的 window.postMessage 传递工具执行和完整状态。启用 Shell MCP 或 web_fetch 后，这个边界问题会放大为本机命令执行、私有记忆泄漏或内网 URL 读取风险。

亮点：当前 `npm run ci:quality` 已通过 actionlint、生产依赖 audit、prompt freeze、TypeScript compile、automation/MCP/Shell/PoW smoke、三浏览器 build/zip、manifest policy 和 release asset check；MCP/Shell 也有真实 smoke，发布资产检查比普通扩展项目更完整。

优先修复：先收紧 window bridge 和 SYNC_STATE 暴露，再把 Shell/web_fetch 这类高风险能力改成显式、可审计的授权边界；随后补关键解析/桥接/同步的单元测试，并拆分 content/fetch-hook/background 的状态域。

总评：B / 5.9。工程基础和 release gate 比较扎实，但安全边界和测试真实性不足，现状不适合把 Shell/MCP 能力默认视作稳定公开能力。

🔍 详细发现

1. 页面世界可以伪造工具执行 bridge 消息

High Confirmed
📁 Security 📍 entrypoints/content.ts:161-189, 1624-1669; entrypoints/main-world.content.ts:134-154 🎯 main-world/content bridge
问题：content script 只校验 `event.origin === window.location.origin` 和 `event.data.source === 'deepseek-pp-main'`，随后对 `EXECUTE_TOOL_CALL` 直接调用 `executeToolCall`，最终进入 `chrome.runtime.sendMessage({ type: 'EXECUTE_TOOL_CALL' })`。同一页面上的任意脚本都可以伪造这个 source/type 组合。
entrypoints/content.ts:161-189 接收 source=deepseek-pp-main 的 EXECUTE_TOOL_CALL 并执行。 entrypoints/content.ts:1624-1669 将 ToolCall 发送到 background EXECUTE_TOOL_CALL。 entrypoints/main-world.content.ts:134-154 也使用同一 window.postMessage 通道发起工具调用。
MINIMAL FIX
不要从页面可伪造的 window message 直接触发工具执行。将工具执行请求改成 content script 自己从已验证的响应流/状态机产生，或引入不可被页面监听/重放的扩展内部通道；所有来自页面世界的执行请求默认拒绝。
⏱ 预计工作量：2-4 天  ·  回归测试：构造页面脚本 `window.postMessage({source:'deepseek-pp-main', type:'EXECUTE_TOOL_CALL', ...})`，断言 background 没有收到任何 EXECUTE_TOOL_CALL；再用真实 DeepSeek SSE 工具调用验证正常路径仍执行。

2. SYNC_STATE 向页面广播完整 memories/skills/preset

High Confirmed
📁 Security 📍 entrypoints/content.ts:1277-1300; entrypoints/main-world.content.ts:55-69 🎯 状态同步
问题：`syncToMainWorld` 用 `window.postMessage` 广播 `memories`、`skills`、`activePreset`、`modelType`、`toolDescriptors`。这不是点对点传输，页面世界的脚本可以监听到完整长期记忆、自定义 Skill 指令和系统 prompt preset，而不仅是本次请求实际注入的内容。
entrypoints/content.ts:1292-1300: window.postMessage({ source:'deepseek-pp-content', type:'SYNC_STATE', memories, skills, activePreset, modelType, toolDescriptors }) entrypoints/main-world.content.ts:55-69: main-world 监听同一广播并 updateHookState。
MINIMAL FIX
不要广播完整状态。content script 应只把本次请求需要的最小 augmentation 结果传给 main-world hook，或把 prompt augmentation 移回隔离世界并让 main-world 只处理 fetch patch 所需的最小字段。
⏱ 预计工作量：2-3 天  ·  回归测试：在页面世界注册 message listener，加载扩展后断言收不到原始 memories/skills/preset；发送消息时验证 prompt augmentation 仍包含被选择的必要上下文。

3. Shell MCP 预设默认 auto/all，缺少逐次确认

High Confirmed
📁 Security 📍 core/shell/policy.ts:9-38; entrypoints/sidepanel/pages/McpPage.tsx:165-180; packages/shell-host/native/shell-mcp-host.mjs:214-340 🎯 Shell Native Host
问题：侧边栏创建 Shell MCP 预设时调用 `createShellMcpPresetInput()`，默认 `enabled: true`、`allowlist.mode: all`、`execution.mode: auto`。native host 的 `shell_exec` 接收任意 command 并通过默认 shell 执行。即使这是产品能力，高风险本机命令也不应在模型输出层默认自动执行。
core/shell/policy.ts:14-37: enabled 默认 true，allowlist all，execution auto。 entrypoints/sidepanel/pages/McpPage.tsx:177-180: 创建 Shell preset 时未传 enabled:false。 packages/shell-host/native/shell-mcp-host.mjs:214-230, 252-340: shell_exec 直接执行 args.command。
MINIMAL FIX
Shell preset 创建后默认 manual 或 disabled；shell_exec 强制逐次用户确认，或至少将默认 allowlist 限制为 shell_status。只有用户明确升级策略后才允许 auto shell_exec。
⏱ 预计工作量：1-2 天  ·  回归测试：创建 Shell preset 后断言 `execution.mode !== 'auto'` 或 `shell_exec` 未注入；模拟 shell_exec 输出时断言需要 UI approval 才调用 native host。

4. 关键路径没有 unit/component test，只有 smoke gate

Medium Confirmed
📁 Testing 📍 package.json:10-40; git tracked test search 🎯 测试结构
问题：`package.json` 没有 `test` 脚本，也没有 Vitest/Jest/Playwright 依赖；tracked files 中没有 `*.test.ts(x)` / `*.spec.ts(x)` / `__tests__`。现有 ci:quality 依赖 smoke 脚本，能证明部分集成路径可运行，但无法稳定覆盖 postMessage bridge、prompt injection、SSE patch、WebDAV schema、Markdown sanitize 等精细边界。
package.json:31: ci:quality 串联 smoke/verify/build，但没有 test runner。 find tracked test files: 无输出。 已验证 npm run ci:quality 通过，但它没有执行常规单元/组件测试。
MINIMAL FIX
引入 Vitest + jsdom，先覆盖 5 个高风险纯函数/桥接边界：tool-parser、fetch-hook sanitization、postMessage gate、WebDAV parse schema、renderMarkdown link sanitizer。
⏱ 预计工作量：1-3 天  ·  回归测试：新增 `npm test` 并加入 ci:quality，确保上述边界测试在 60 秒内完成。

5. 手动 release 的 tag 输入没有绑定 checkout ref

Medium Confirmed
📁 Release 📍 .github/workflows/release.yml:7-12, 27-30, 60-91, 116-141 🎯 Release workflow
问题：workflow_dispatch 接收 `tag` 输入，但 checkout 没有用该 tag/ref；后续脚本只用 `INPUT_TAG` 做版本匹配、npm package 版本和 GitHub Release 名称。手动触发时可能构建当前 workflow ref 的源码，却创建或更新输入 tag 对应的 release。
release.yml:7-12 定义 tag input。 release.yml:27-30 checkout 未设置 ref。 release.yml:61-73 和 117-141 使用 INPUT_TAG 派生 VERSION/TITLE/gh release，但没有校验 `git rev-parse $TAG == $GITHUB_SHA`。
MINIMAL FIX
workflow_dispatch 时 checkout `ref: ${{ inputs.tag }}`；发布前校验 tag 存在、当前 HEAD 等于 tag object/commit，并把该校验纳入 release quality gate。
⏱ 预计工作量：2-4 小时  ·  回归测试：用 workflow dry-run 或本地脚本模拟 input tag 与当前 HEAD 不一致，断言 release job fail fast。

6. WebDAV/本地导入只强转数组，没有 schema 校验

Medium Confirmed
📁 Stability 📍 entrypoints/background.ts:642-657, 945-957; SettingsPage.tsx:365-380; memory/store.ts:68-72 🎯 持久化导入边界
问题：WebDAV download 只确认远端 JSON 是数组，然后 `return parsed as T[]` 并 replaceAll；本地 memory import 也把 JSON 直接断言为 `Memory[]`。这会让畸形对象进入 IndexedDB/chrome.storage，后续 UI、prompt injection 或 memory selector 可能读到缺字段/错类型数据。
entrypoints/background.ts:945-957: parseRemoteArray 只做 JSON parse + Array.isArray。 entrypoints/background.ts:648-653: replaceAllMemories/replaceAllCustomSkills/replaceAllSkillSources/replaceAllPresets。 entrypoints/sidepanel/pages/SettingsPage.tsx:374-377: JSON.parse(text) as Memory[] 后逐条 SAVE_MEMORY。
MINIMAL FIX
为 Memory/Skill/SystemPromptPreset/GitHubSkillSource 增加运行时 schema validator；下载或导入发现任一 item 不合法时停止并展示具体文件、索引和字段错误，不写入任何数据。
⏱ 预计工作量：1-2 天  ·  回归测试：用缺少 `content`、`tags` 非数组、`createdAt` 非数字的 memories.json 做下载/import，断言本地数据未被替换且错误信息包含索引。

7. inline agent Markdown renderer 可生成不安全链接属性

Medium Confirmed
📁 Security 📍 entrypoints/content.ts:864-910 🎯 renderMarkdown
问题：`appendInlineAgentFinalAnswer` 用 `innerHTML = renderMarkdown(text)` 插入模型生成文本。renderer 先转义 `< > &`，但链接替换直接把 `$2` 放入 `href="$2"`，没有 URL 协议白名单，也没有转义引号。模型输出的 Markdown 链接可生成 `javascript:` URL 或属性注入。
entrypoints/content.ts:870: textDiv.innerHTML = renderMarkdown(text)。 entrypoints/content.ts:884-910: link regex 输出 `$1`，未校验协议/未转义 href。
MINIMAL FIX
不要用字符串拼接生成链接。解析 Markdown link 后创建 HTMLAnchorElement，href 只允许 http/https/mailto，文本使用 textContent；或者使用带 sanitizer 配置的成熟 Markdown renderer。
⏱ 预计工作量：3-6 小时  ·  回归测试：输入 `[x](javascript:alert(1))` 和 `[x](" onclick="alert(1))`，断言输出 DOM 不含 javascript href 或额外事件属性。

8. MCP HTTP/SSE 原始响应读取没有字节上限

Medium Confirmed
📁 Performance 📍 core/mcp/transports/common.ts:85-141; core/mcp/client.ts:264-290 🎯 MCP transport
问题：MCP result 的 `maxResultBytes` 只在 `normalizeMcpToolResult` 阶段截断展示文本；HTTP transport 在此之前对非 SSE 调用 `response.text()`，SSE path 也把 chunks 累加到 `buffer`。恶意或异常 MCP server 可以在超时前返回大 body，绕过展示层限制造成内存压力。
core/mcp/transports/common.ts:102: const raw = await response.text()。 core/mcp/transports/common.ts:128-130: buffer += decoder.decode(value, { stream: true })。 core/mcp/client.ts:274-276: maxResultBytes 只在 rendered.length 后截断 detail。
MINIMAL FIX
在 transport 读取层加入 `maxResponseBytes`，流式累计字节超过上限立即 cancel reader 并返回 mcp_response_too_large；SSE buffer 也按字节/字符双上限裁剪。
⏱ 预计工作量：4-8 小时  ·  回归测试：mock MCP server 返回超过 limits.maxResultBytes 10 倍的 JSON/SSE body，断言 transport fail fast 且不会进入 normalizeMcpToolResult。

9. content/fetch-hook/background/McpPage 形成大文件多职责

Medium Confirmed
📁 Maintainability 📍 entrypoints/content.ts; core/interceptor/fetch-hook.ts; entrypoints/background.ts; McpPage.tsx 🎯 前端/扩展核心
问题：多个关键文件超过 1000 行并混合多个状态域。`content.ts` 同时负责桥接、工具卡片、inline agent trace、token speed、主题/background、宠物拖拽和 DOM 清理；`fetch-hook.ts` 同时 hook fetch/XHR/IndexedDB、prompt patch、SSE 过滤和工具执行；background switch 覆盖 memory/skill/MCP/WebDAV/automation/chat。
wc -l: entrypoints/content.ts 3534 行，core/interceptor/fetch-hook.ts 1245 行，entrypoints/background.ts 1101 行，entrypoints/sidepanel/pages/McpPage.tsx 1296 行。 content.ts function inventory 超过 100 个函数，含 pet/token/inline-agent/tool-cleanup/background 多个职责域。
MINIMAL FIX
按状态域拆分：bridge-security、tool-rendering、inline-agent-trace、token-speed、pet、theme/background、dom-cleanup。每次修复一个 finding 时先抽对应模块和测试，不做一次性大重写。
⏱ 预计工作量：1-2 周渐进拆分  ·  回归测试：为拆出的每个模块加最小行为测试，并用 ci:quality 确认 prompt freeze、MCP/Shell smoke、build:all 仍通过。

10. memory_save 对无效 payload 静默补默认值并持久化

Medium Confirmed
📁 Fallback 📍 core/tool/memory.ts:164-174 🎯 Memory tool runtime
问题：tool schema 要求 `type/name/content/tags`，但 runtime 没有 fail fast。缺失 name 会保存为 `unnamed`，非法 type 会保存为 `topic`，缺失 tags 会保存空数组，缺失 content 会保存空字符串。这会把模型格式错误或恶意桥接调用转成持久数据污染。
core/tool/memory.ts:166-173: name = stringValue(payload.name) || 'unnamed'; type = memoryTypeValue(payload.type) || 'topic'; content = stringValue(payload.content); tags = stringArrayValue(payload.tags);
MINIMAL FIX
memory_save/update/delete 在 runtime 层按 descriptor schema 验证 payload；必填字段缺失或空字符串时返回 `memory_invalid_payload`，不写 DB。
⏱ 预计工作量：4-8 小时  ·  回归测试：调用 memory_save 缺 name/content/tags 或 type=bad，断言返回失败且 `getAllMemories()` 数量不变。

11. Shell tag 清理疑似可能隐藏未执行工具文本

Low Suspected
📁 Fallback 📍 entrypoints/content.ts:1323-1328 🎯 DOM cleanup
问题：`buildToolTagPattern` 总是把 `SHELL_TOOL_NAMES` 加入可清理 tag 模式，即使当前 toolDescriptors 中没有 shell_exec/shell_status。fetch-hook 执行路径不会执行未注入的 Shell 工具，但 DOM cleanup 仍可能清掉可见 `` 文本，让用户误以为模型没有输出工具调用或无法看到失败原因。
entrypoints/content.ts:1323-1328: const names = new Set(catalogNames); for (const name of SHELL_TOOL_NAMES) names.add(name); return escaped.length > 0 ? escaped.join('|') : 'memory_save|memory_update|memory_delete';
MINIMAL FIX
只清理当前已注入、可执行或已确认执行过的 tool tag；未在 descriptor 中的 shell tag 应保留或替换为明确的“工具不可用”提示。
⏱ 预计工作量：2-4 小时  ·  回归测试：在无 Shell descriptor 状态下渲染包含 `{...}` 的回复，断言文本不会被静默移除。

12. OfficeCLI style library eager 加载增加 background 体积

Info Confirmed
📁 Performance 📍 core/skill/officecli-library.ts:4-32, 71-84, 187-196; dist/chrome-mv3/background.js 🎯 Dependency weight
问题：OfficeCLI official skills 和 52 个 style docs 使用 eager glob，在模块加载时组装 style library。当前 build 可接受，但 background.js 已约 870KB，扩展总包约 3.05MB；这部分内容属于按需使用的 Skill 资料，没必要成为 background 初始化成本。
core/skill/officecli-library.ts:4-32: 多个 import.meta.glob(... eager:true)。 core/skill/officecli-library.ts:77-84: officialStyleLibrary/morphReferences 在模块加载阶段生成。 npm run ci:quality build 输出：dist/chrome-mv3/background.js 870.38 kB，Total size 3.05 MB。
MINIMAL FIX
把 OfficeCLI style details 改成按需加载或延迟组装；列表页只加载 index/metadata，用户真正启用 `/officecli-styles` 或相关 Skill 时再加载完整 style docs。
⏱ 预计工作量：1-2 天  ·  回归测试：build 后比较 background.js 体积；加载 Skill library 时验证 OfficeCLI style skill 仍能完整生成。
